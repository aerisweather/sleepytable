Intro

SleepyTable exists to fill the need of a lightweight extendable jQuery plugin for dealing with RESTful JSON APIs. At HAMweather we have needed easy to use tables for several projects to interact with our various APIs. The SleepyTable has a remote data and paging core, with extensions that enable sorting, filtering, and pager (visual buttons) that interact with the core to request data from a REST service. Paging, sorting, and filtering are handled by the server resource. The plugin is designed to be minimally invasive by adding only the Javascript global SleepyTable and all classes that the table uses are namespaced with sleepytable-

Properties

General
Name Type Default Description
debug boolean false The main debug flag for the plugin. (SleepyTable's plugins also have their own debug flag)
theme string "default" A theme name that is applied to the end of the table class. The <table> tag will have a class sleepytable-[theme] applied to it so different tables on a single page may be styled differently.
widthFixed (Not Implemented) boolean false Fix the width of columns between pages
heightFixed boolean true Row height between pages is fixed. This helps avoid the parent page hopping around as the user changes pages.

Server/Ajax Settings
Name Type Default Description
url string "http://www.example.com/api/?page={page}&limit={limit}" The url of the API to consume. The url may have tokens {page}, {limit}, {offset} that correspond to the page, limit, and offset of data, respectively. The paging may also be done with HATEOAS data (see below).
ajaxProcessing function null A function that is called when data is returned from the server. This can be used to transform server data into the page data structure. Default parsing is provided by using the data-src attribute of the table head but will be overridden if this function is set.
rawRowDataSelection string null Where the actual row data is in the repsonse. Useful if you're not using the ajaxProcessing function or the data-src attribute of the headers. For example: If you don't know the headers of data (or do not want to use data-src attributes) but would like to just load in data, you may use this path to describe where the row data is.
ajaxTransport function $.ajax The function that is used for the data call to the server. It should be a jQuery Ajax compatible call. (I'm not really sure what others use here. If you have a use case that doesn't fit quite right, let me know on GitHub)
dataType string "jsonp" The requested data type of the response. "jsonp" is used for remote data sources where usually you can use "json" for local, same domain requests.
httpMethod string "GET" The HTTP Method used to retrieve data from the API.
traditionalSerialization boolean false "Set this to true if you wish to use the traditional style of param serialization." - jQuery Manual for $.ajax()
currentPageNumber integer 1 The initial page of data the table should load.
page0Based boolean false Whether the first page the API starts on 0 or 1. Computer people generally like page 0 as a start, but traditional paging for APIs starts at 1. Set this based on whatever type of person designed the API you will be using.
offsetInitial integer 0 Initial records to skip.
limit integer 10 How many records to fetch initially.
hateoas object null An object defining paging methods and where the data may be found in a HATEOAS repsonse. HATEOAS data may be supplied by the API you are consuming so you may easily traverse pages by merely sending the "next page" to a URI specified in the first repsonse.
If an API response looks something like: 

{
	success  : "true",
	errors   : null,
	response : {
		[data]
	},
	links   : {
		first: "http://www.example.com/api/?sort=id:-1,page=1",
		previous: "http://www.example.com/api/?sort=id:-1,page=2",
		next: "http://www.example.com/api/?sort=id:-1,page=3",
		last: "http://www.example.com/api/?sort=id:-1,page=20"
	}
}
Then paging can be handled with a hateoas option like this: 

hateoas: {
	pageFirst    : 'links.first',
	pagePrevious : 'links.previous',
	pageNext     : 'links.next',
	pageLast     : 'links.last'
}
preLoadNextPage boolean true Load the next page right away so users don't have to wait for the server/network.
offsetInitial integer 0 Initial records to skip.
silentDelay integer (milliseconds) 500 How long to wait for user interaction to stop before requesting data. For example: Let a user click multiple headers to get their sort options set before sending a request.

Plugins Just on how to load them, more docs on individual plugins below
Name Type Default Description
plugins array ['pluginName', 'pluginName2',...] [] An array of plugin names to load. Order will be preserved for all callbacks.
pluginOptions Object {pluginName:{ <Plugin Options> }, pluginName2: {...}, ...} {} Where settings for individual plugins reside. An object who's property names correspond with plugin names containing their own objects of settings for individual plugins.

Templates
Name Type Default Description
loadingRowHtml string (of HTML) '<tr class="sleepytable-loading"><td colspan="9999"><img src="/js/jquery.sleepytable/loading.gif"/> Loading...</td></tr>' A row is added for loading once all the data rows are removed. This loading row is displayed until there is a response from the server with new data.
emptyRowHtml string (of HTML) '<tr class="sleepytable-empty"><td colspan="9999">[No entries]</td></tr>' A row is added if no data is returned from the server for the requested criteria. This row is removed once a query returns data.

Events

Events are triggered throughout SleepyTable's operation. To quickly use an event, a user can pass a function named on[Event] as a property to the SleepyTable object, then that function will be called just after the plugins have called the specified event. There is a "tester" plugin included that logs the events to the console. Feel free to load the plugin and add "tester" to the plugins configuration array.

Name Arguments Description
init SleepyTableObj Run after the core properties have been set for the table. No data requests have been made yet. Best for setup of plugins.
headersChanged SleepyTableObj, Headers (a jQuery array of th elements) If the headers are loaded dynamically from the data, this event gets called when we learn what the headers are. Commonly used to just re-init a plugin that deals with headers.
getFetchParams SleepyTableObj, Data (Object of existing data) Called when a new request is made to the server. Must add/remove properties from the Data param and return it. Sorting uses this method to compile and add the sort variable to the request string.
pageLoading SleepyTableObj This is called when the table goes into "Loading" mode. All rows are cleared from the table and a special loading message is displayed. This happens when some of the query params (other than paging) change.
pageLoaded SleepyTableObj, pageData Fired when a new page is loaded (no necessarily displayed)
pageDisplay SleepyTableObj, pageNumber, pageObj, pageRaw (raw data response from server) Called when a page is displayed to the end user after rows are parsed.
clearedPages SleepyTableObj Fired when the page cache is cleared an new query data needs to be sent to the server. Usually used in resetting the plugin for initial paging.

Methods

Some functions are publicly available for calling. Something like $("#table").SleepyTable('method', arg1, arg2, ...); can be used to call methods.

Name Arguments Description
fetchPage pageNumber Fetches the specified page into the object. Does not go to the page.
setPage pageNumber Sets the plugin's page to the given page and renders it.
clearPages Clears the internal cache of pages, fetches first page. Useful for when query parameters change
first Jump to the first page.
previous Jump to the previous page, if available.
next Jump to the next page, if available.
last Jump to the last page, if available.

Plugins

SleepyTable was designed to be extendable. Page handling with a remote source is really the only core feature, the sorting, filtering and paging buttons are all plugins that interact with the core. We have created events needed for these plugins but events in other places may be applicable as well.

Sort

Sorting allows the headers of the table to be clicked allowing for sorting information to be passed back to the server resulting in server side sorted data.

Filter

The filter plugin allows users to search for rows in a table. This data is passed back to the server for server side sorting.

Pager

The pager plugin is a simple plugin that binds paging events to buttons. These buttons are smart enough to go into a disabled state when they are no longer applicable. This plugin also can display current page information and select how many rows are displayed at once.
